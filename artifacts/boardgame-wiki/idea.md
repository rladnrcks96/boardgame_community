# 보드게임 위키 & 커뮤니티

## Problem Statement

보드게임 정보 탐색 여정이 나무위키(문서)·BGG(랭킹/메타데이터)·네이버카페·디스코드(커뮤니티)로 흩어져 있는 지금, "이 게임 사도 될까? 룰이 뭐지? 같이 할 사람 있나?"라는 하나의 질문에 한 곳에서 답하게 만들 수 있을까?

## Recommended Direction

마이크로 위키 + 통합 커뮤니티(C안)로 시작한다. 인기 게임 20-30개를 골라 본인이 직접 3-5줄 룰 요약을 써서 콜드스타트를 해결하고, 그 게임 문서 위에 바로 별점 리뷰와 게시판(공략/변형룰/모임구인)을 붙인다. 별도 시스템 3개(위키/커뮤니티/업적)를 순차로 쌓는 원안 로드맵 대신, 위키와 커뮤니티라는 두 가정을 하나의 얇은 화면에서 동시에 검증한다.

차별화는 신규 역량이 아니라 "한국어로 룰+리뷰+구인을 한 화면에서 보게 하는 UX 통합"에서 나온다는 점을 인정하고 시작한다. 이게 약점이 아니라, 검증해야 할 지점이다.

업적/뱃지는 코드 구현 자체는 쉽지만, 포인트→뱃지 전환 규칙(특히 안티스팸: 지연 지급, 되돌리기 반영)은 실제 편집 행동 데이터 없이는 설계해도 검증할 수 없다. 그래서 `activities` 로그 적재는 지금부터 시작하지만, 규칙 확정과 UI 노출은 실사용 데이터가 쌓인 뒤로 미룬다.

## Key Assumptions to Validate

- [ ] Must — 보드게임 유저가 룰+리뷰+게시판을 "한 화면"에서 보고 싶어한다 (검증: 본인이 시딩한 20-30개 게임 문서를 지인/커뮤니티에 공유하고 재방문·체류시간 확인)
- [ ] Must — 본인이 직접 쓴 20-30개 문서만으로도 첫 방문자가 "쓸만하다"고 느낀다 (검증: 신규 방문자에게 "이걸로 모임 준비 가능한가" 직접 물어보기)
- [ ] Should — 로그인 후 진입장벽 없이 바로 편집 가능해야 나무위키/디스코드 대신 여기서 쓴다 (검증: 편집 버튼 클릭 → 실제 저장까지 완료율 측정)
- [ ] Should — 카테고리 게시판(공략/변형룰/모임구인)이 실제 모임 성사로 이어진다 (검증: 모임구인 글의 응답률·성사 여부 추적)
- [ ] Might — 업적/뱃지가 참여를 늘린다 (지금 검증 안 함, activities 로그만 적재)
- [ ] Might — BGG 랭킹 그리드가 메인 전환에 중요하다 (지금 검증 안 함)

## MVP Scope

**포함**
- 게임 상세 페이지: BGG XML API로 20-30개 인기 게임 메타데이터 시딩 (이름/카테고리/인원수/플레이타임/난이도/이미지는 hotlink)
- 위키 본문: 본인이 직접 작성한 3-5줄 룰 요약, 마크다운 저장, 로그인 후 즉시 편집 가능, 버전 이력(`game_revisions`) 저장 (diff UI는 제외)
- 별점 리뷰 (`reviews`)
- 카테고리 게시판 — 처음부터 공략/변형룰/모임구인 3개 카테고리로 분리 (`posts`/`comments`), 댓글, 좋아요
- Supabase Auth 로그인/가입
- 메인 페이지: BGG 글로벌 랭크 기준 그리드
- `activities` 로그 테이블 — 문서 편집/리뷰/댓글/좋아요 이벤트 적재 시작 (포인트 계산·뱃지·UI 노출은 없음)

**제외** (Not Doing 참고)

## Not Doing (and Why)

- 뱃지 자동 지급·리더보드 UI — 포인트/안티스팸 규칙이 아직 실사용 데이터 없이는 검증 불가. 로그만 쌓고 규칙은 나중에 소급 적용
- 편집 지연 지급·되돌리기 반영 로직 — 스팸 편집 패턴을 실제로 봐야 규칙을 설계할 수 있음
- 자체 평점으로 랭킹 소스 전환 — 자체 리뷰가 쌓이기 전까지 의미 없음, BGG 랭킹과 병기만
- 위키 버전 diff 비교 화면 — 버전 이력은 저장하되(유실 방지), 비교 UI는 나중
- 다국어 지원 — 한국어 단일로 시작
- 룰북 원문 재게시 — 저작권 문제로 처음부터 커뮤니티 재작성 방식만 허용, 공식 PDF는 링크만

## Open Questions (해결됨)

- **BGG API 이미지/데이터 사용** — XML API2는 "광고·구독 수익을 주목적으로 하는 사용"을 상업적 사용으로 간주해 금지하고, 상업적 사용 시 별도 라이선스가 필요하다 ([Terms of Use](https://boardgamegeek.com/wiki/page/XML_API_Terms_of_Use), [Commercial Use 안내](https://boardgamegeek.com/wiki/page/BGG_XML_API_Commercial_Use)). 지금은 광고 없는 비상업적 사이드 프로젝트이므로 이미지 hotlink 포함 API 사용에 문제없음. **나중에 광고/구독 모델을 붙이면 그 시점에 BGG에 커머셜 라이선스 문의 필요** — 이 조건을 로드맵에 남겨둔다
- **초기 시딩 20-30개 게임 기준** — BGG 글로벌 랭킹 상위로 선정. 본인이 잘 아는 게임이 아니므로 룰 요약 작성 시 시간이 더 걸릴 수 있음을 감안한다
- **게시판 카테고리** — 처음부터 공략/변형룰/모임구인 3개로 분리한다. 구현 부담이 크지 않고, 모임구인처럼 시간에 민감한 글이 공략처럼 오래 남는 글과 한 목록에 섞이면 처음부터 UX가 헷갈리기 때문
